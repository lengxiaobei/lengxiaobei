# 代码审查模板

当需要审查代码时，使用以下模板进行评价。

## 评价维度

从以下维度评价代码（每项 0-100 分，加权平均为总分）：

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| 语法正确性 | 25% | 无 SyntaxError、导入正确 |
| 代码风格 | 10% | PEP 8、命名规范 |
| 性能 | 15% | 时间复杂度、资源使用 |
| 安全性 | 15% | 注入风险、敏感信息 |
| 架构 | 15% | 模块划分、依赖方向 |
| 可读性 | 10% | 命名清晰、结构清楚 |
| 可维护性 | 5% | 易于修改和扩展 |
| 可测试性 | 5% | 易于编写测试 |

## 严重程度分级

| 级别 | 描述 | 扣分 |
|------|------|------|
| trivial | 轻微问题（格式、命名） | -2 |
| minor | 次要问题（可读性） | -5 |
| major | 主要问题（性能、架构） | -15 |
| critical | 严重问题（语法错误、安全漏洞） | -30 |

## 审查流程

1. 静态分析：AST 语法检查、复杂度计算、未使用导入检测
2. LLM 深度审查：发送代码 + 模板，获取评分和问题列表
3. 综合评分：静态分析扣分 + LLM 评分 = 最终评分
4. 判定：存在 critical 问题或 ≥3 个 major 问题 → 需要重新生成

## 输出格式

返回 JSON：
```json
{
  "overall_score": 85,
  "issues": [
    {
      "category": "performance",
      "severity": "major",
      "description": "嵌套循环导致 O(n²) 复杂度",
      "line_number": 42,
      "suggestion": "使用字典缓存避免重复计算"
    }
  ],
  "improvement_suggestions": ["建议1", "建议2"],
  "needs_revision": false
}
```
